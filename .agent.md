# vertex-platform — Instrucciones Operativas del Agente

Este archivo define los protocolos de calidad, reglas arquitectónicas y recordatorios
obligatorios que el agente debe aplicar en toda sesión de trabajo sobre este repositorio.

---

## Identidad del Proyecto

- **Repo:** `vertex-platform` — capa de orquestación del SaaS multi-tenant
- **Template aislado:** `ecommerce-vertex` — NO modificar salvo actualización global explícita
- **Backend:** Firebase Functions v2 · TypeScript strict-mode
- **Frontend:** Angular 17+ standalone · Signals · `ChangeDetectionStrategy.OnPush`
- **Entornos:** `vertex-platform-dev` (dev) · `vertex-platform-app` (prod)

---

## Version Management & Development Workflow

### Protocolo Obligatorio de Back-Merge (Sincronización Inversa)

**TRIGGER:** Esta instrucción se activa en los siguientes eventos:
1. Se completa una tarea que actualiza la versión en `package.json` o en `provisioning.ts` (`CURRENT_TEMPLATE_VERSION`)
2. Se completa cualquier tarea que requiera un merge de `develop` → `main`
3. Se aplica un hotfix sobre la rama `main`
4. Se interactúa con la estructura de despliegue (CI/CD, `firebase.json`, `.github/workflows/`)

**ACCIÓN REQUERIDA DEL AGENTE:** Al completar cualquiera de los eventos anteriores, recordar
explícitamente al usuario que ejecute el siguiente bloque antes de iniciar cualquier nueva funcionalidad:

```bash
# Flujo Obligatorio de Sincronización Inversa (Back-Merge)
git checkout develop
git pull origin develop
git merge origin/main
# Resolver conflictos manteniendo la estabilidad si los hubiera
git push origin develop
```

**Criterio de éxito verificable:**
```bash
git diff develop..main   # No debe devolver ninguna salida de código fuente
```

**Razón de la regla:** El pipeline de aprovisionamiento multi-tenant genera commits automáticos
(versiones en `package.json`, `CURRENT_TEMPLATE_VERSION` en `provisioning.ts`). Una divergencia
entre `develop` y `main` puede causar regresiones silenciosas de versión en tenants ya desplegados.

### Estrategia de ramas

```
feature/* → develop → main
                ↑
         (Back-Merge obligatorio e inmediato)
```

| Rama | Propósito |
|------|-----------|
| `main` | Producción. CI/CD automático a `vertex-platform-app` |
| `develop` | Integración. Base de todo trabajo nuevo |
| `feature/*` | Funcionalidades. Desde `develop`, merge a `develop` |
| `hotfix/*` | Parches urgentes. Desde `main`, merge a `main` Y a `develop` |

## Constraints Arquitectónicos Activos

### Hot-Configuration Principle
- La configuración de negocio de cada tenant (nombre, colores, credenciales MP) **NUNCA** va
  en el build del template. Siempre se escribe en `settings/storeConfig` del Firestore del
  subproyecto GCP via Firestore REST API autenticada con `getOwnerOAuthClient()`.

### Aislamiento de Template
- **NO tocar** `ecommerce-vertex` a menos que el usuario solicite explícitamente una
  actualización global de versión.
- Toda lógica de negocio, seeding y configuración va en `vertex-platform/functions/`.

### firebase-admin Privilege Loop
- Las mutaciones cross-project usan `getOwnerOAuthClient()` → `apiFetch()`.
- Nunca usar el SDK de Admin de Firebase directamente para escribir en proyectos hijo.
- El acceso a Secret Manager usa `SecretManagerServiceClient` del proyecto `PLATFORM_PROJECT`.

### TypeScript Strict
- Todos los archivos en `functions/src/` deben compilar con `strict: true`.
- Tipar explícitamente todos los retornos de funciones públicas.
- Usar `unknown` en lugar de `any` en catch blocks; castear solo tras validación.

### Error Resiliency
- Toda operación GCP/Firebase en `functions/src/` debe estar dentro de try/catch.
- Usar `retry()` de `helpers.ts` para operaciones susceptibles a latencia de API.
- Loguear con `console.error()`/`console.warn()` antes de relanzar.
- Mensajes de error sanitizados antes de enviarse al cliente vía `HttpsError`.

## Recordatorios de Seguridad

- `localhost` como dominio autorizado en Firebase Auth: **solo en `vertex-platform-dev`**.
  Nunca agregar `localhost` a `vertex-platform-app` (producción).
- Custom claims (`platformAdmin: true`): solo se asignan vía `admin.ts` con privilegio de
  superadmin. No exponer endpoints para auto-asignación de claims.
- CORS: restringido estrictamente a los dominios registrados en `ALLOWED_ORIGINS` de `helpers.ts`.

---

## Seeding Engine

- Los verticales disponibles son: `indumentaria`, `gastronomia`, `retail`.
- El flag `includeMockData` controla la inclusión de clientes y órdenes simuladas.
- El customizador `customizeSeed` reemplaza referencias a `'Vertex'` con el nombre del tenant.
- **Nunca hardcodear** el nombre comercial del tenant en los seeds. Siempre parametrizado.
